# 实时数仓 - 运行程序选择指南

## 📋 三种运行程序说明

本项目提供了三个 Java 主程序，用于不同的测试场景：

---

## 1️⃣ OdsDataReader.java（推荐首选）

### 用途
- **最简单**的测试程序
- 仅测试 ODS 层数据读取
- 验证 MySQL CDC 能否正常工作

### 特点
- ✅ 代码简单，易于理解
- ✅ 只读取 MySQL CDC 并打印
- ✅ 不涉及复杂的表创建
- ✅ 适合快速验证数据源连接

### 何时使用
- 第一次运行时
- 测试 MySQL CDC 连接
- 验证 Kafka 配置

### 运行命令
```bash
mvn exec:java -Dexec.mainClass="com.example.rtwarehouse.OdsDataReader"
```

### 预期输出
```
MySQL CDC Source> {"before":null,"after":{"id":1,"user_id":123,...},...}
```

---

## 2️⃣ WarehouseLayerTest.java（推荐逐步测试）

### 用途
- **分层测试**程序
- 按 ODS -> DIM -> DWD -> DWS -> ADS 顺序加载
- 每层加载后会暂停提示

### 特点
- ✅ 逐步验证各层 SQL
- ✅ 容易定位问题在哪一层
- ✅ 带有详细的日志输出
- ✅ 适合调试 SQL 语法问题

### 何时使用
- ODS 测试通过后
- 需要验证某层 SQL 是否正确
- 排查特定层的错误

### 运行命令
```bash
mvn exec:java -Dexec.mainClass="com.example.rtwarehouse.WarehouseLayerTest"
```

### 预期输出
```
========================================
实时数仓分层测试开始
========================================

========== 测试 ODS 层 ==========
加载 ODS 层 SQL...
✓ ODS 层加载成功

ODS 层测试完成，是否继续测试下一层？(y/n)

========== 测试 DIM 层 ==========
加载 DIM 层 SQL...
✓ DIM 层加载成功

... (依次类推)
```

---

## 3️⃣ RealtimeWarehouseApplication.java（完整运行）

### 用途
- **生产环境**完整运行
- 一次性加载所有层
- 完整的实时数仓作业

### 特点
- ✅ 完整的数仓流程
- ✅ 包含所有层级
- ✅ 会一直运行直到手动停止
- ✅ 适合正式运行和性能测试

### 何时使用
- 所有测试都通过后
- 需要正式运行数仓
- 生产环境部署前的本地测试

### 运行命令

## 🎯 推荐的测试流程

### 阶段 1: 环境验证（第 1 天）

#### Step 1: 检查 SQL 文件
```bash
powershell -ExecutionPolicy Bypass -File .\check_sql_files.ps1
```

#### Step 2: 测试 ODS 层
```bash
mvn exec:java -Dexec.mainClass="com.example.rtwarehouse.OdsDataReader"
```
**目标**: 能看到 MySQL CDC 数据流入

**预计耗时**: 5-10 分钟

---

### 阶段 2: 分层测试（第 2 天）

#### Step 3: 运行分层测试
```bash
mvn exec:java -Dexec.mainClass="com.example.rtwarehouse.WarehouseLayerTest"
```

**目标**: 
- 验证所有层 SQL 语法正确
- 确认各层表都能创建成功
- 检查是否有字段类型或约束问题

**预计耗时**: 30-60 分钟

**可能遇到的问题**:
- SQL 语法错误 → 查看具体报错信息
- 字段类型不匹配 → 检查源表和目标表定义
- Kafka Topic 创建失败 → 检查 Kafka 配置

---

### 阶段 3: 完整运行（第 3 天及以后）

#### Step 4: 运行完整版
```bash
mvn exec:java -Dexec.mainClass="com.example.rtwarehouse.RealtimeWarehouseApplication"
```

**目标**:
- 验证端到端数据流
- 检查数据在各层的转换
- 监控性能和资源使用

**预计耗时**: 持续运行

**监控要点**:
- Checkpoint 是否正常
- 各 Task 的反压情况
- 数据延迟是否在可接受范围

## 🔧 常见问题处理

### 问题 1: 编译失败

**症状**: `mvn clean package` 失败

**解决**:
```bash
# 清理 Maven 缓存
mvn clean

# 重新下载依赖
mvn dependency:purge-local-repository

# 再次编译
mvn clean package -DskipTests
```

### 问题 2: ClassNotFound 错误

**症状**: 运行时找不到某个类

**解决**:
```bash
# 确保先编译
mvn clean package -DskipTests

# 使用 mvn exec 运行，不要直接用 java -jar
mvn exec:java -Dexec.mainClass="..."
```

### 问题 3: 内存不足

**症状**: OutOfMemoryError

**解决**: 编辑 `pom.xml`，在 exec-maven-plugin 配置中添加：
```xml
<configuration>
    <systemProperties>
        <systemProperty>
            <key>org.apache.flink.kubernetes.memory.process.main.size</key>
            <value>4096m</value>
        </systemProperty>
    </systemProperties>
</configuration>
```

## 💡 最佳实践建议

### 1. 本地开发
- 设置 parallelism = 1
- 关闭 checkpoint 或使用较长间隔
- 使用本地 MySQL 和 Kafka

### 2. 测试环境
- parallelism = 2
- checkpoint 间隔 60 秒
- 使用测试数据库

### 3. 生产环境
- parallelism >= 4（根据数据量）
- checkpoint 间隔 30 秒
- 启用 savepoint
- 配置监控告警
